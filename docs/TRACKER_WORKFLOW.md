# Tracker Workflow

## Базовый процесс

1. Пользователь создаёт задачу в трекере.
2. Исполнитель берёт задачу в работу и выполняет её.
3. Если задача слишком крупная или неоднородная, она декомпозируется на более мелкие подзадачи.
4. Каждая задача должна описывать целостную, логически завершённую фичу или подфичу.
5. Фичи не должны быть слишком объёмными.
6. Если фича получается слишком большой, вместо одной задачи создаётся крупная `parent issue`, внутри которой заводятся `sub-issues`.

## Правила декомпозиции

- Задача должна давать понятный и проверяемый результат.
- Задача должна быть достаточно маленькой, чтобы её можно было реализовать, проверить и отревьюить без избыточного контекста.
- Если для завершения фичи нужно заметно менять backend, frontend, данные и UX одновременно, это кандидат в крупную `parent issue`.
- Если задачу сложно описать одним коротким результатом в формате "после выполнения пользователь может ...", задачу нужно делить.

## Что считать задачей

Подходящая задача:

- одна пользовательская возможность;
- одна техническая подфича, без которой следующая пользовательская возможность не может быть завершена;
- один законченный инфраструктурный или платформенный шаг, если он открывает дальнейшую разработку.

Неподходящая задача:

- "сделать весь frontend";
- "реализовать весь движок заданий";
- "добавить всю аналитику";
- любая работа, которая естественным образом раскладывается на несколько независимых результатов.

## Что считать parent issue

Крупная `parent issue` создаётся, если:

- фича состоит из нескольких независимых пользовательских сценариев;
- результат нельзя выпустить одним компактным изменением;
- для завершения потребуется серия задач, которые логично выполнять и принимать по отдельности.

## Шаблон для задач

- Название: короткое, предметное, без технического шума.
- Результат: что станет возможным после выполнения.
- Критерии готовности: 2-5 проверяемых пунктов.
- Ограничения: что в задачу не входит, чтобы не раздувать scope.

## Шаблон для parent issue

- Название: крупная пользовательская или платформенная область.
- Цель: какой законченный блок продукта строится.
- Список `sub-issues`: каждая задача даёт самостоятельный проверяемый результат.
- Граница: что явно остаётся вне рамок этой `parent issue`.
